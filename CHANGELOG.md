# Changelog — форк sem0007/ccbot

Отличия нашего форка от upstream `six-ddc/ccbot` (база — коммит `9d7ab97`).
Пиши сюда КАЖДУЮ правку перед пушем: дата, что и зачем. Коротко.

Термины: «воркер» — окно tmux с запущенным `claude`; «тема» — форум-топик
Telegram; «сессия» — сессия Claude Code (файл `.jsonl`).

## 2026-07-10 — отказоустойчивость, локальный API, тема-пульт, багфиксы

**Единое ядро.** `src/ccbot/service.py` — класс `ControlService`: вся логика
управления (создать/послать/продолжить/убить/список/сверка) реализована один
раз, транспортно-независимо. Telegram-обработчики и HTTP-API — тонкие обёртки.
`MessageBus` рассылает ответы Claude веером в Telegram И в кольцевой буфер для
API (человек и агент видят один поток).

**Локальный HTTP-API.** `src/ccbot/api.py` (`aiohttp`), слушает `127.0.0.1:8787`.
Bearer-авторизация fail-closed (без `CCBOT_API_TOKEN` сервер не стартует),
постоянное сравнение токена, аудит-лог, allowlist каталогов для создания
сессий. ~18 ручек (`/health`, `/sessions`, `/sessions/{id}/send|resume|output|
screenshot|messages|restart|heal`, `/bindings`, `/monitor`, `/resumable`,
`/reconcile`, `/control-topic`, `/logs`). Это стенд отладки: ALLOWED_USERS
пускает к боту только человека, поэтому агент водит бота через этот API.

**Три бага сессий/продолжения закрыты структурно:**
- *resume не находил сессию* — поиск возобновляемых сессий теперь идёт glob-ом
  по всем проектам со сверкой реального `cwd` из `.jsonl` (не слепо по имени
  папки); при resume подхватывается фактический `session_id` (Claude может его
  форкнуть).
- *новая сессия не привязывалась к теме* — `thread_id/user_id/chat_id` передаются
  ЯВНО (не из перетираемого `user_data`); флаг `pending_bind` защищает свежее
  пустое окно от очистки; смещение монитора предрегистрируется (новая→0, чтобы
  отдать вводный ответ; resume→размер файла, чтобы не переигрывать историю).
- *ответы не доходили в тему* — самочинящийся `find_users_for_session`
  (перечитать map + подхватить реальный `session_id`) и проактивный посев
  `group_chat_id`.

**Отказоустойчивость.** `reconcile()` — идемпотентная сверка
tmux↔session_map↔state↔monitor_state (на старте, периодически, по `/reconcile`).
`supervise()` перезапускает упавшие фоновые задачи с backoff и пределом.
Центральный `error_handler`. Запись `state.json` под `asyncio.Lock` +
пакетная транзакция. Бот переведён под **systemd user-service** (`deploy/
ccbot.service`, `Restart=always`, linger); воркеры живут в tmux-сессии `ccbot`
и переживают рестарт бота, `reconcile` их усыновляет.

**Тема-пульт.** `src/ccbot/handlers/control_topic.py` — постоянная непривязанная
форум-тема с дашбордом (статус сессий) и кнопками обновить/сверка/убить. Работает
и в личке бота (Bot API 9.3, темы в личных чатах): `CCBOT_CONTROL_CHAT_ID` = id
пользователя, супергруппа не нужна. Перед переиспользованием проверяется живость
темы (если удалена — создаётся заново, без дублей). Зарегистрирована реальная
команда `/kill` (раньше рекламировалась, но уходила мусором в Claude).

**Хук.** `hook.py`: регэксп UUID сделан регистронезависимым.

**Конфиг (`config.py`).** Новые переменные окружения (все в `~/.ccbot/.env`, НЕ
в репозитории): `CCBOT_API_TOKEN/HOST/PORT`, `CCBOT_API_ALLOWED_CWDS`,
`CCBOT_CONTROL_CHAT_ID/TOPIC_NAME`, `CCBOT_RECONCILE_INTERVAL`. `CCBOT_API_TOKEN`
добавлен в список вычищаемых секретов (не утекает в дочерние `claude`).

**Временное.** `bot.py`: диагностический логгер входящих обновлений (`DBG_UPDATE`,
группа −1) — убрать после разбора проблемы «первое сообщение в новой теме».

**Документация процесса.** Добавлены `CHANGELOG.md`, `docs/DEVELOPING.md` (цикл
edit→push→deploy, правило «пушить с записью в changelog») и `scripts/
check-secrets.sh` — обязательный страж утечки секретов/приватных данных перед
пушем (читает реальные значения из `~/.ccbot/.env` и ищет их в изменениях +
типовые шаблоны). Усилён `.gitignore` (state/логи/ключи).
